## Architecture
- **Navigation**: RootNavigator (`src/navigation/RootNavigator.tsx`) renders AuthStack or TabNavigator based on session. TabNavigator has bottom tabs; Templates tab has a nested native stack (list -> detail -> exercise picker). Type params: `AuthStackParamList` in RootNavigator, `TemplatesStackParamList` in TabNavigator.
- **Auth** (`src/contexts/AuthContext.tsx`): Session via `supabase.auth.onAuthStateChange`. `setCurrentUserId()` is called **unconditionally** on every auth event (including `INITIAL_SESSION`/`TOKEN_REFRESHED`) AND from the initial `getSession().then(...)` rehydrate — so the database module's `currentUserId` mirrors the live session at all times, including cold starts with restored sessions. The `newUserId !== prevUserId` guard now only wraps the expensive `resetDatabase() + pull*` block inside `SIGNED_IN` (not the setter). On `SIGNED_IN` with new user ID, sets `syncing=true`, runs `resetDatabase()` (nuclear file-delete + reinit — corruption-proof), pulls data from Supabase, sets `syncing=false`. On `SIGNED_OUT`, the unconditional setter resets to `'local'`. RootNavigator gates on `syncing` to show spinner (prevents WorkoutScreen race). 30s timeout (`SYNC_TIMEOUT_MS`) prevents hung spinner. `useAuth()` returns `{ session, user, loading, syncing }`. Email/password + Google OAuth (expo-web-browser + expo-auth-session, scheme `liftai`). Session in expo-secure-store.
- **Database** (`src/services/database.ts`): expo-sqlite async API. `PRAGMA journal_mode = WAL` + `PRAGMA foreign_keys = ON` (set per-connection in separate `execAsync` before DDL). `DB_NAME = 'workout-enhanced.db'` exported constant. `resetDatabase()` closes connection, deletes file via `SQLite.deleteDatabaseAsync`, reinitializes — uses `dbInitPromise` assignment to prevent concurrent `getDb()` races. `deleteWorkout`, `deleteTemplate`, `clearLocalUpcomingWorkout` wrapped in `withTransactionAsync`. `clearAllLocalData` intentionally NOT transactional (called from concurrent sync flows). Tables: exercises (`user_id` nullable — NULL=global, non-null=custom), user_exercise_notes (`user_id`, `exercise_id`, `form_notes`, `machine_notes`), templates, template_exercises, workouts (with `upcoming_workout_id`, `session_notes`, `coach_notes`, `exercise_coach_notes`, `planned_exercise_ids`), workout_sets (with `target_weight`, `target_reps`, `target_rpe`, `exercise_order`, `programmed_order`), upcoming_workouts, upcoming_workout_exercises, upcoming_workout_sets. All queries use typed row interfaces (not `any`). Centralized row mappers: `parseExercise` (ExerciseRow), `mapWorkoutSetRow` (WorkoutSetRow — tag cast, bool conversion, order default), `parseExerciseFromJoin` (e_-prefixed join columns), `parseExerciseFromTemplateJoin` (exercise_-prefixed join columns), `mapUpcomingWorkoutSetRow` (tag default + cast). Batch ops: `getBulkExercises()`, `addWorkoutSetsBatch()`, `getTemplateExerciseCountsBatch()`. `stampExerciseOrder()` persists exercise sequence at finish time. `setPlannedExerciseIds()`, `getPlannedExerciseIds()`, `insertSkippedPlaceholderSets()` manage plan tracking and ghost rows. All functions have try/catch with Sentry. Uses `safeJsonParse` for muscle_groups. **Module-level userId**: `currentUserId` set by AuthContext via `setCurrentUserId()` on every auth event (including cold-start rehydrate). Call sites (`createExercise`, `upsertExerciseNote`, `getUserExerciseNotes`, `getUserExerciseNotesBatch`) resolve via `resolveUserId()` — returns `currentUserId` if it's a real id, else falls back to `supabase.auth.getSession()` and self-heals the module global (defense in depth against any future write that races ahead of AuthContext propagation). Returns `'local'` only when genuinely no session. **Two note types in user_exercise_notes**: `form_notes` (technique tips, synced with MCP), `machine_notes` (seat/pin settings, private from AI). CRUD: `getUserExerciseNotes()`, `getUserExerciseNotesBatch()`, `upsertExerciseNote()`. Thin wrappers: `updateExerciseFormNotes()`, `updateExerciseMachineNotes()` delegate to `upsertExerciseNote()`.- **Theme**: Dark theme tokens in `src/theme/tokens.ts`, re-exported from `src/theme/index.ts` with `modalStyles` from `src/theme/sharedStyles.ts`. Tokens in separate file to avoid circular deps. `colors.info` (#4A9EFF) for informational badges (e.g., RPE in history).
- **Types**: All DB types in `src/types/database.ts`. Workout UI types (`LocalSet`, `ExerciseBlock`, `PreviousSetData`) in `src/types/workout.ts`. `Exercise` has `user_id: string | null` (NULL=global, non-null=custom) and NO note fields. `ExerciseNotes` = `{ form_notes, machine_notes }`. `ExerciseWithNotes` = `Exercise & ExerciseNotes` (used in callbacks from ExerciseDetailModal). `SetTag` = warmup/working/failure/drop. `TemplateExercise` has `warmup_sets`. `UpcomingWorkoutSet` has optional `target_rpe` and `tag`. `Workout` has `upcoming_workout_id` (links to originating plan), `session_notes` (free-form user notes during workout), `coach_notes` (AI coach workout-level notes from upcoming workout), `exercise_coach_notes` (JSON map `{exercise_id: note}` of per-exercise AI coach notes), `planned_exercise_ids` (JSON array of exercise_ids in plan order at workout start; NULL for ad-hoc empty workouts). `WorkoutSet` has optional `target_weight`, `target_reps`, `target_rpe` (persisted plan targets), `exercise_order` (stamped at insert time, re-stamped at finish and after auto-reorder), `programmed_order` (nullable; plan position stamped at insert, NULL for user-added mid-workout or pre-migration).
- **Constants**: `src/constants/exercise.ts` — MUSCLE_GROUPS, EXERCISE_TYPE_OPTIONS, REST_SECONDS, DEFAULT_REST_SECONDS.
- **Observability**: Sentry (`@sentry/react-native`) in App.tsx. Org: `sachit-goyal`, project: `react-native`. Disabled when `EXPO_PUBLIC_SENTRY_DSN` unset.
- **Error Handling**: `ErrorBoundary` (`src/components/ErrorBoundary.tsx`) wraps AuthProvider. Reports to Sentry, shows recovery UI. All fire-and-forget DB writes (set completion, tag cycling, debounced set changes, session notes, machine notes, widget bridge Live Activity calls) have `.catch(e => Sentry.captureException(e))`. `fireAndForgetSync` uses `.catch(() => {})` (errors already captured internally by `handleSyncError`). `liveActivity.ts` bare catch blocks report to Sentry except `cancelTimerEndNotification` (expected failure when notification already fired).
- **Live Activity** (`src/services/liveActivity.ts`): Persistent iOS Live Activity for entire workout. Read-only widget shows exercise name + set counter always, rest timer (countdown + progress bar) conditionally. All functions async (return `Promise<void>`), no-op on Android. Deep link: `liftai://workout`. `startWorkoutActivity` is **idempotent** — updates existing activity if one exists, creates new only when `currentActivityId` is null or dead (dismissed by iOS). Safe to call from multiple code paths (fresh start, resume-on-focus, first-exercise-add) without stacking widgets. Rest timer end notification shows banner with "Rest Complete" title and plays sound (`timeSensitive` interruption level). All notification ops are serialized via `serializedNotificationOp` to prevent cancel/schedule races (errors routed to Sentry). `updateWorkoutActivityForRest` takes `(exerciseName, totalSeconds, setNumber, totalSets)` — encodes rest duration in subtitle as `"Set X/Y|D"` where D = max rest seconds (parsed by Swift widget for progress bar proportional display). `stopRestTimerActivity` cancels notification above the `!currentActivityId` guard so it always runs. Module-level `currentMaxRestSeconds` tracks the progress bar denominator (increases on +15s, never decreases on -15s). `applyPendingWidgetActions()` reads Swift widget intent actions (`liftai_action_queue` key) from UserDefaults on foreground return — syncs +/-15s adjustments and skip-rest from lock screen buttons into RN timer state.
- **Lock Screen Widget** (`plugins/withInteractiveLiveActivity/`): iOS 17+ read-only Live Activity. `UnifiedWorkoutView` shows exercise name + set counter header (identical in both states), large countdown timer + purple progress bar during rest. No interactive buttons. `ParsedSetState` parses `"Set X/Y|D"` pipe-format subtitle to extract `totalRestSeconds` for progress bar proportional display. Rest state shown conditionally via `timerEndDateInMilliseconds > 0`. Timer `Text` and `ProgressView` use `.id(restEndTime)` to force SwiftUI recreation on adjustments. Plugin must run AFTER `expo-live-activity` in `app.config.ts`. `NEW_SWIFT_FILES = []` — `WorkoutIntents.swift` and `WorkoutUserDefaultsHelper.swift` exist on disk but are excluded from widget target.
- **Workout Bridge** (`src/services/workoutBridge.ts`): RN -> iOS widget state sync via App Groups UserDefaults. `syncStateToWidget` writes `WidgetState`, `clearWidgetState` removes it. All no-op on Android. No polling — widget is read-only.
- **Hooks** (`src/hooks/`): Extracted from WorkoutScreen for testability and separation of concerns. `useRestTimer` — manages rest countdown interval, AppState foreground resync, vibration, and Live Activity updates; exposes `isResting`/`currentEndTime`/`startRestTimer`/`adjustRestTimer`/`dismissRest`; takes `onRestEnd`/`onRestUpdate` callbacks for widget sync; `onRestUpdate` receives `(isResting, endTime, exerciseName?)` — exercise name threaded through to widget bridge for correct lock screen display during rest; `endingRef` prevents re-entrant endRest calls (fixes multiple vibrations); foreground resync polls `applyPendingWidgetActions()` before computing remaining time (syncs Swift widget +/-15s and skip actions); then uses `endRest(false)` if expired (no vibrate — notification already alerted user). `useWidgetBridge` — builds widget state from exercise blocks (with `preferBlockIdx` + `lastActiveBlockRef` for active exercise tracking), syncs to UserDefaults + Live Activity; `buildWidgetState` and `syncWidgetState` accept optional `restingExerciseName` parameter — during rest, overrides the "next incomplete set" search to show the correct exercise the user is resting from (fixes wrong exercise name after auto-reorder). `useNotesDebounce` — debounced machine notes persistence via `updateExerciseMachineNotes()` (exercise-level only, no set-level write). `useExerciseBlocks` — owns `exerciseBlocks` state, `blocksRef`, `originalBestE1RMRef`, `currentBestE1RMRef` (always-in-sync PR comparison ref), `prSetIdsRef`, `pendingSetWritesRef` (debounced DB writes, 300ms per set ID, flushed on finish/unmount, cleared on cancel), and all block-mutation handlers (`handleSetChange`, `handleCycleTag`, `handleAddSet`, `handleDeleteSet`, `handleToggleMachineNotes`, `handleToggleRestTimer`, `handleAdjustExerciseRest`, `handleMachineNotesChange`, `handleRemoveExercise`); uses internal `updateBlock`/`updateBlockSets` helpers to eliminate repeated immutable-update boilerplate. `useSetCompletion` — owns `handleToggleComplete` with single coalesced `setExerciseBlocks` call (completion + auto-reorder + PR bestE1RM in one updater — was 3 separate calls), `validationErrors` state, and `reorderToast` state; pre-computes reorder decision and PR result before the single state update; reads bestE1RM from `currentBestE1RMRef` (always in sync, unlike `blocksRef` which lags by one render). **Double-tap guard**: `pendingCompletionRef` Map tracks in-flight toggles by set ID; second tap to same direction bails out; guard self-clears when `blocksRef` catches up, also released on validation failure. `useWorkoutLifecycle` — owns all lifecycle state (loading, activeWorkout, templateName, templates, upcoming workout, finish modal, summary, add-exercise modal state, history modal) and handlers (loadState, start/finish/cancel workout, add exercise, template preview, session notes). **`workoutNotesRef`** mirrors `workoutNotes` state — `confirmFinish` reads from ref to avoid stale closure (user types + taps Finish within 500ms debounce). **`confirmFinish`** reads `blocksRef.current` instead of `exerciseBlocks` closure for set-order stamping, counts, and template update plan. **Focus skip**: `loadState` skips `loadActiveWorkout` on tab re-focus if same workout ID is already loaded (`prevWorkoutId === active.id` check before overwriting `workoutRef`), preventing in-flight debounced edits from being clobbered. `getExerciseHistoryData` extracted to `src/utils/exerciseHistory.ts` as a standalone utility used by both `useExerciseBlocks` and `useWorkoutLifecycle`. Shared refs (`workoutRef`, `blocksRef`) created at WorkoutScreen component level and passed to hooks to avoid circular dependencies.
- **Shared UserDefaults** (`modules/shared-user-defaults/`): Local Expo module for App Groups read/write. Group ID: `group.com.sachitgoyal.liftai`.
- **1RM Estimation Engine** (`src/utils/oneRepMax.ts`): Two-path engine for estimated 1RM. **Path A** (RPE provided): Tuchscherer-style percentage table lookup (`src/data/rpeTable.ts`, 12 rows x 9 RPE columns with bilinear interpolation). **Path B** (no RPE): Rep-range-weighted ensemble of Epley + Brzycki + Wathen formulas. Types in `src/types/oneRepMax.ts`. Exports: `calculateE1RM()` (returns `E1RMResult` with confidence tier), `calculateEstimated1RM()` (backwards-compat number), `getPRGatingMargin()`, `FRESHNESS_HALF_LIFE_DAYS` (42 days). **Confidence tiers**: HIGH (1 rep, or 1-5 reps with RPE >= 7, margin 0-3%), MEDIUM (6-10 reps, margin 6%), LOW (11+ reps, margin 12%). **PR gating**: WorkoutScreen uses confidence-gated thresholds — HIGH: any improvement counts, MEDIUM: must beat by 1%, LOW: must beat by 3%. Database: `getBestE1RM()` (all-time), `getCurrentE1RM()` (freshness-weighted decay), `getE1RMWithConfidence()` (with confidence tier).
- **Utilities** (`src/utils/`): `exerciseSearch.ts` — `filterExercises()` by name/muscle group. `streakCalculation.ts` — `calculateStreak()` counts consecutive workout days. `setTagUtils.ts` — `getSetTagLabel()`/`getSetTagColor()` for tag display. `exerciseTypeColor.ts` — `exerciseTypeColor()` maps ExerciseType to theme color. `format.ts` — `formatDuration()`/`formatDate()`. `uuid.ts` — UUID generation.
- **Environment**: `app.config.ts` (dynamic Expo config). `.env.development` / `.env.production` for Supabase separation (gitignored). Dev Supabase: `lift-ai-dev` (ref: `gcpnqpqqwcwvyzoivolp`). Prod: `lift.ai` (ref: `lgnkxjiqzsqiwrqrsxww`). EAS builds use env vars inlined in `eas.json`.

## Screens
All screens in `src/screens/`. Key non-obvious behaviors:

### WorkoutScreen (~550 lines)
Orchestrates hooks and renders UI. Logic in `useExerciseBlocks`, `useSetCompletion`, `useWorkoutLifecycle`. UI in `ExerciseBlockItem`, `WorkoutTimers` (`RestTimerBar`/`ElapsedTimer`), `WorkoutIdleScreen` (`NoActiveWorkout`), `WorkoutSummary`. Styles in `WorkoutScreen.styles.ts`.

- **Set row layout**: 5 columns (SET, LBS, REPS, RPE, Checkbox) — no dedicated PREV column. Previous data as gray placeholders (`rgba(107, 107, 114, 0.5)`), upcoming targets as purple placeholders (`rgba(124, 92, 252, 0.45)`). Inputs are borderless background pills (surfaceLight on surface).
- **Debounced writes**: Set input changes (weight/reps/RPE) debounced 300ms per set ID to SQLite, flushed on finish/unmount, cleared on cancel. Checkbox toggling persists immediately.
- **Auto-fill on completion**: Fills empty weight/reps from upcoming targets (priority 1) or previous session (priority 2). RPE from targets only. User-entered values preserved.
- **RPE rules**: Warmup sets hide RPE (empty). Failure sets hide RPE (inherently RPE 10, stored as null).
- **Tags**: Tap set number to cycle (working -> warmup W -> failure F -> drop D). Long-press or swipe-left to delete set.
- **Warmup styling**: Uncompleted warmup sets have amber tint (`colors.warningBg`); completed green overrides.
- **Validation**: Completion requires weight + reps (red background). Finish requires 1+ completed sets.
- **Rest timer**: Auto-starts on set completion when enabled (per-exercise seconds from template, default 150s).
- **PR detection**: On set completion, if e1RM beats all-time best (cached as `bestE1RM` on `ExerciseBlock`) past confidence-gated threshold (HIGH: 0%, MEDIUM: 1%, LOW: 3%), shows "PR" badge + double-tap haptic. Uses `calculateE1RM()` + `getPRGatingMargin()`. `originalBestE1RMRef` + `currentBestE1RMRef` caches avoid re-fetching on uncheck/recheck.
- **Auto-reorder**: First set completion for out-of-position exercise moves it to top of incomplete blocks (below completed) with `LayoutAnimation`. Only when `prevCompletedCount === 0`. Un-completing doesn't trigger. PR update uses exercise ID lookup (not blockIdx) for post-reorder safety. Persists the new order to DB via `stampExerciseOrder` so reload preserves the reordered sequence.
- **Machine notes**: Inline in workout (toggle below sets). Persists to `exercises.machine_notes` via `useNotesDebounce` (500ms debounce → `updateExerciseMachineNotes()` → fire-and-forget sync). Private from AI coach.
- **Coach tips**: Collapsible purple-tinted section per block, shown when starting from upcoming workout with MCP per-exercise notes.
- **Session notes**: Free-form multiline below "Add Exercise". Stored in `workouts.session_notes`. 500ms debounced save via `updateWorkoutSessionNotes()`. Visible to MCP AI coach. Distinct from exercise-level notes (permanent) and coach tips (MCP-generated).
- **Planned vs Actual**: Starting from upcoming workout persists `upcoming_workout_id`, `planned_exercise_ids`, and writes `target_weight`/`target_reps`/`target_rpe` per set, so comparisons survive completion and sync. `handleStartFromTemplate` / `handleStartFromUpcoming` persist the exercise_id list via `setPlannedExerciseIds` **before** the `Promise.all(buildExerciseBlock)` to close the race window where an interrupted start could leave `planned_exercise_ids = NULL`. `handleStartEmpty` writes no plan (column defaults to NULL). `handleAddExerciseToWorkout` does NOT mutate `planned_exercise_ids`; it computes the next `exercise_order` as `Math.max(...existingOrders) + 1` (not `length + 1`) so it can't collide with a gap left by a prior `handleRemoveExercise`. `handleAddSet` inherits the block's `exercise_order` so every set in a block shares the block's sequence position. `LocalSet` carries `exercise_order` in memory so `blocksRef` can read it without a DB roundtrip. `confirmFinish` dedups `planned_exercise_ids` before inserting ghost rows (first-occurrence position wins).
- **Template update (F5)**: On finish with template-based workout and modified set counts (F3) or exercise order (F2), summary shows "Template Changes Detected" + "Update Template" button. `buildTemplateUpdatePlan()`/`computeOrderDiff()` in `setDiff.ts`. Plan computed eagerly in `confirmFinish()` before blocks cleared. `applyWorkoutChangesToTemplate()` applies atomically. Before stamping performed order, reads `planned_exercise_ids` and inserts ghost rows (via `insertSkippedPlaceholderSets`) for any planned exercise not in the live blocks, so the AI coach can detect planned-but-skipped exercises.
- **Live set tracking (F3)**: `ExerciseBlock` has `originalWarmupSets`/`originalWorkingSets` stamped at start. `computeSetDiffs()`/`hasSetChanges()` in `setDiff.ts` compare current vs originals for F5 prompt. Counts restored on resume via template lookup.
- **Last performed badge**: Template cards on idle screen show relative date ("Today"/"3 days ago") via `getLastPerformedByTemplate()` + `formatLastPerformed()`.
- **Performance**: Single coalesced `setExerciseBlocks` in `handleToggleComplete`. Self-contained `React.memo` timer components (own intervals, no parent re-render). Lazy-mounted modals. `FlatList` for add-exercise. Pre-computed `targetMap`/`errorSet` via `useMemo`. Custom `areEqual` on `ExerciseBlockItem`. Parallelized DB queries via `Promise.all`. Module-level constant objects. `prSetIds` via ref (not state).

## Supabase Sync (`src/services/sync.ts`)
- `syncToSupabase()` — push custom exercises (user_id IS NOT NULL, no note columns), user_exercise_notes separately, templates, workouts + sets to Supabase (auth-guarded, adds user_id). Global exercises (user_id=NULL) are NOT pushed. Includes `upcoming_workout_id`, `session_notes`, `coach_notes`, `exercise_coach_notes`, and `planned_exercise_ids` on workouts and `target_weight`/`target_reps`/`target_rpe`/`exercise_order`/`programmed_order` on workout sets. **Self-healing rescue block** runs at the top (after `getDb()`, before any SELECT): `DELETE FROM user_exercise_notes WHERE user_id = <session> AND exercise_id IN (SELECT exercise_id FROM user_exercise_notes WHERE user_id = 'local')` (local row wins the composite-PK collision), then `UPDATE user_exercise_notes SET user_id = <session> WHERE user_id = 'local'`, then `UPDATE exercises SET user_id = <session> WHERE user_id = 'local'`. Wrapped in its own try/catch routing to `handleSyncError('rescue local rows', err)` so a rescue failure can't abort downstream pushes. Rescues any row that slipped through under the default `'local'` id before the session-filtered push queries would otherwise skip it.
- `deleteTemplateFromSupabase(id)` / `deleteTemplateExerciseFromSupabase(id)` — fire-and-forget deletes, never throw, errors to Sentry
- `pullExercisesAndTemplates()` — pulls all exercises (global + custom via RLS, no user_id filter), upserts without note columns. Separately pulls `user_exercise_notes` and upserts. Syncs rest_seconds/warmup_sets bidirectionally. Deletes remotely-removed template_exercises. Batch-fetches with `.in()`.
- `pullWorkoutHistory()` — upserts finished workouts + sets. Converts `is_completed` bool->int for SQLite.
- `pullUpcomingWorkout()` — pulls latest upcoming workout + exercises + sets
- **Important**: Do NOT wrap sync pull loops in `withTransactionAsync` — they run concurrently via Promise.all, SQLite can't handle concurrent transactions
- **Sync triggers**: login (clear -> pull all), workout finish (push), WorkoutScreen focus (pull all sequentially), template/exercise delete (fire-and-forget push), template exercise stepper edits (fire-and-forget push), exercise notes edit (fire-and-forget push after debounce)
- All console.log/error are `__DEV__`-guarded (prod uses Sentry only)

## Gotchas
- **SQLite corruption recovery**: If phone dies mid-workout, WAL checkpoint can corrupt the DB file. `clearAllLocalData()` (DELETE FROM) silently fails on corrupted files. Fix: logout/login triggers `resetDatabase()` which deletes the file and reinitializes. All data recovers from Supabase. **In-progress un-synced workouts are lost** (sync only pushes finished workouts). `db` typed as `SQLiteDatabase | undefined`; `resetDatabase` assigns `dbInitPromise` synchronously to prevent concurrent `getDb()` races.
- **Supabase migrations**: When adding columns referenced by sync or MCP, run the migration on both dev (`gcpnqpqqwcwvyzoivolp`) and prod (`lgnkxjiqzsqiwrqrsxww`) via SQL Editor before deploying code. Migration files go in `supabase/migrations/`.
- Alert.prompt is iOS-only; use Modal or Alert.alert for cross-platform.
- metro.config.js adds COOP/COEP headers for expo-sqlite OPFS VFS on web.
- `withInteractiveLiveActivity` plugin must run AFTER `expo-live-activity` in plugins array.
- Machine notes are permanent and synced: `flushPendingNotes()` on finish, `clearPendingNotes()` on cancel (discard doesn't overwrite stored notes). 500ms debounce via `debouncedSaveNotes`. Fire-and-forget `syncToSupabase()` after each debounced save. Notes debouncing logic extracted to `src/hooks/useNotesDebounce.ts` hook (with cleanup on unmount). Both note types (`form_notes`, `machine_notes`) live in `user_exercise_notes` table (not on exercises): `form_notes` (technique, synced with MCP), `machine_notes` (private from AI, shown inline in workout). `ExerciseDetailModal` loads notes via `getUserExerciseNotes()` on open, passes `ExerciseWithNotes` to `onExerciseUpdated` callbacks. `useWorkoutLifecycle` loads machine notes via `getUserExerciseNotes()`/`getUserExerciseNotesBatch()` when building exercise blocks.
- **Do NOT build to device from git worktrees.** Worktrees lack `.env.*` files (gitignored), and Expo CLI has Metro URL/device discovery issues with non-standard paths. Always merge to main and build from `/Users/sachitgoyal/code/lift-ai/`.
- **Plugin Swift edits require clean prebuild.** When editing Swift files in `plugins/withInteractiveLiveActivity/swift/`, you must run `npx expo prebuild --clean` before building — `expo run:ios` won't re-copy plugin files if `ios/` already exists.
- **Rowid is no longer a sort tiebreaker in `getWorkoutSets`.** `ORDER BY exercise_order, set_number` depends on `exercise_order` being stamped at every insert path: `buildExerciseBlock` stamps it from the plan index, `handleAddExerciseToWorkout` stamps `Math.max(...existingOrders) + 1`, `handleAddSet` inherits the block's order. Any NEW insert path must also stamp `exercise_order` — default-zero rows will sort to the top and break visual order.
- **Ghost rows are NOT identified by `exercise_order IS NULL`.** Migration 007 declared the column `NOT NULL DEFAULT 0` and the constraint is preserved. Ghost rows (planned-but-skipped exercises written at finish) use the composite sentinel: `programmed_order IS NOT NULL AND exercise_order = 0 AND is_completed = 0 AND reps = 0 AND weight = 0`. See `supabase/migrations/013_workout_ordering_integrity.sql` for the documented rule. MCP `get_workout_detail` filters ghosts by this same heuristic (empty `sets` after excluding rows that match).
